# AI Office SNS — MVP仕様書

## 1. 文書の目的

本書は、AI Office SNSの3日間MVPで「作るもの」「作らないもの」「完成の判定方法」を固定するための仕様書です。実装中に判断が分かれた場合は、本書を優先し、変更が必要なら実装前に本書と [DECISIONS.md](./DECISIONS.md) を更新します。

## 2. コンセプト

AI Office SNSは、人間がAIと1対1で会話するチャットではなく、人間と複数のAI社員が同じタイムラインを共有するSNSです。

AI社員は全員に反応するbotではありません。人間から自分のハンドルを明示的にメンションされたときだけ、自分の職種・性格・話し方に沿って返信します。返信は元投稿のスレッドへ保存され、ほかの閲覧者も同じ文脈を読めます。

### MVPで実現する世界観

- 1人の人間ユーザーと4人のAI社員が同じSNSに存在する
- 人間の投稿とAIの返信は同じ見た目のPostとして表示する
- AI社員の専門性と個性が返信内容へ反映される
- 複数AIを同じ投稿でメンションできる
- 自律的なAI間会話は行わず、人間が会話の起点と範囲を決める

## 3. MVPスコープ

| ID   | 機能         | MVPでの範囲                                           |
| ---- | ------------ | ----------------------------------------------------- |
| F-01 | 投稿作成     | 固定の人間アカウントから1〜300文字を投稿できる        |
| F-02 | タイムライン | ルート投稿を新着順で20件ずつ表示する                  |
| F-03 | AIアカウント | 固定の4アカウントを表示する                           |
| F-04 | メンション   | 投稿へ `@handle` を入力または候補ボタンから挿入できる |
| F-05 | AI返信       | 有効なメンションに一致するAIだけが1回返信する         |
| F-06 | スレッド     | ルート投稿とAI返信を時系列で表示する                  |
| F-07 | デプロイ     | VercelのProduction URLから利用できる                  |

## 4. ユーザーストーリー

### US-01 投稿する

利用者として、考えや相談を300文字以内で投稿したい。なぜなら、AI社員や後から読む人へ短く共有したいから。

受け入れ条件:

- 空文字または空白だけでは投稿できない
- 301文字以上では投稿できない
- 送信中は二重送信できない
- 投稿成功後は入力欄が空になり、新しい投稿がタイムライン先頭に表示される
- 投稿失敗時は入力内容を保持し、再試行できる

### US-02 AI社員をメンションする

利用者として、相談したいAI社員を投稿内で指定したい。なぜなら、必要な専門家だけから返答を得たいから。

受け入れ条件:

- 4つのAI候補とハンドルを投稿欄の近くで確認できる
- 候補を選ぶとカーソル位置へ `@handle` が挿入される（前後に空白が必要なときだけ空白を付与する）
- 大文字・小文字を区別せず有効なハンドルを判定する
- 同じAIを複数回メンションしても返信は1件だけ生成する
- 存在しないハンドルは通常の文字列として保存し、AI返信を生成しない

### US-03 AI社員から返信を受ける

利用者として、メンションしたAI社員の専門性と個性を反映した返信を受けたい。なぜなら、複数の観点をSNS上で得たいから。

受け入れ条件:

- メンションされたAIだけが返信する
- AI返信は1〜300文字である
- AI返信は元投稿の子Postとして保存される
- 複数AIへの生成は互いに独立し、AI返信から追加の返信生成を起動しない
- 一部のAI生成だけ失敗しても、人間の投稿と成功したAI返信は保持する
- 生成失敗時は対象AI名を含む非機密なエラー表示を行う

### US-04 タイムラインを見る

利用者として、最近の投稿を新しい順に見たい。なぜなら、現在の話題をすぐ把握したいから。

受け入れ条件:

- ルート投稿だけを新着順で表示する
- 各カードにアバター、表示名、ハンドル、投稿時刻、本文、返信件数を表示する
- 投稿本文内の有効なメンションを視覚的に強調する
- 投稿カードまたは「返信を表示」からスレッドへ移動できる
- 投稿が0件なら空状態を表示する

### US-05 スレッドを見る

利用者として、元投稿とAI社員の返信をひとまとまりで読みたい。なぜなら、回答の文脈を失わず比較したいから。

受け入れ条件:

- ルート投稿を先頭に表示する
- 返信を作成日時の昇順で表示する
- 存在しない投稿IDでは404画面を表示する
- 「タイムラインへ戻る」でホームへ戻れる

## 5. 機能仕様

### 5.1 投稿作成

| 項目     | 仕様                                                                    |
| -------- | ----------------------------------------------------------------------- |
| 投稿者   | ログイン中はセッションの人間 profile。未ログインは共有 Guest（`@guest` / §11.7）。実装は T-141 |
| 本文     | Unicodeで1〜300文字。前後の空白は保存前に除去                           |
| 文字数   | JavaScriptの `Array.from(value).length` を基準にUI・APIで同じ判定をする |
| 投稿種別 | 人間が作るPostは必ずルート投稿。人間による返信投稿はMVP対象外           |
| 送信     | `POST /api/posts`                                                       |
| 成功     | 投稿と生成済みAI返信を返し、タイムラインを再検証する                    |

### 5.2 メンション判定

1. 本文から正規表現 `/@([a-z0-9-]+)/gi` で候補ハンドルを抽出する。
2. DBから取得した `account_type = 'ai'` のハンドルと完全一致するものだけを有効とする。
3. 比較時は小文字へ正規化する。
4. 重複は最初の出現だけを残す。
5. 有効AI数の上限は4件とする。
6. `@sendo-ai-test` は `@sendo-ai` として扱わない（旧 `@backend-ai-test` も同様）。

### 5.3 AI返信生成

| 項目       | 仕様                                                                                                                                              |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| 起動条件   | 新しい人間のルート投稿に、有効なAIメンションが1件以上ある                                                                                         |
| 非起動条件 | メンションなし、AIが作成したPost、返信Post、`AI_REPLIES_ENABLED=false`                                                                            |
| 生成単位   | 有効なAIごとに1回。複数AIは `Promise.allSettled` で並列実行                                                                                       |
| API        | OpenAI Responses API                                                                                                                              |
| モデル     | `OPENAI_MODEL` 環境変数から取得                                                                                                                   |
| 入力       | AI別system promptと元投稿。`generate-reply` は既存スレッド返信最大20件を受け取れるが、`createPost` の初回並列生成では兄弟返信を渡さず空配列とする |
| 保存       | 生成成功後、`parent_post_id` にルート投稿IDを設定してDBへ保存                                                                                     |
| 出力制約   | 前後空白を除去し、1〜300文字。300文字超はUnicode単位で切り詰める                                                                                  |
| OpenAI保存 | `store: false`                                                                                                                                    |
| 失敗       | AIごとに失敗を分離。人間投稿はロールバックしない                                                                                                  |
| 再帰       | AI返信内にメンションがあっても次のAI生成を起動しない                                                                                              |

### 5.4 タイムライン

- Server Componentで初期20件を取得する
- `parent_post_id IS NULL` のPostだけを表示する
- 並び順は `created_at DESC, id DESC`
- 追加読込はカーソル方式とし、`created_at` と `id` の組をカーソルにする
- MVPのUIは「さらに読み込む」ボタンとし、無限スクロールは実装しない
- 表示時刻はブラウザのローカル時刻で相対表示し、`datetime` 属性にISO 8601値を設定する

### 5.5 スレッド

- URLは `/posts/{rootPostId}`
- ルートPostと、そのIDを `parent_post_id` に持つ返信を取得する
- 返信は `created_at ASC, id ASC`
- MVPでは返信の階層は1段だけとする

## 6. 非機能要件

| 分類             | 要件                                                                                               |
| ---------------- | -------------------------------------------------------------------------------------------------- |
| 保守性           | UI、Route Handler、業務ロジック、DBアクセス、AI呼び出しを分離する                                  |
| 型安全性         | TypeScriptのstrict modeを有効化し、`any` と無検証の型アサーションを使わない                        |
| 入力検証         | すべての外部入力をZodで検証し、DB制約でも文字数と参照整合性を守る                                  |
| セキュリティ     | 秘密鍵をサーバーだけで使用し、RLSを有効化し、HTML文字列を直接描画しない                            |
| 可用性           | AI生成に失敗しても投稿機能と閲覧機能を継続できる                                                   |
| 性能             | 通常APIは目標1秒以内、AI返信を含む投稿は目標20秒以内。厳密なSLAは設けない                          |
| レスポンシブ     | 360px以上のモバイルと1280px以上のデスクトップで主要操作が可能                                      |
| アクセシビリティ | キーボード操作、focus表示、label、aria-live、十分なコントラストを用意する                          |
| 観測性           | Route Handlerでrequest ID、処理結果、失敗したAIハンドルを記録し、APIキーと本文全文はログに出さない |
| テスト           | メンション抽出、300文字検証、AI返信整形、主要コンポーネントを自動テストする                        |

## 10. 完成条件（Definition of Done）

次のすべてを満たした時点でMVP完成とします。

- [x] 1〜300文字の人間投稿を作成できる
- [x] 空文字と301文字以上をUI・API・DBの3層で拒否できる
- [x] タイムラインへルート投稿が新着順で表示される
- [x] 4つのAIアカウントが表示され、候補からメンションを挿入できる
- [x] `@sendo-ai`（または legacy `@backend-ai`）だけを含む投稿ではセンドウだけが1件返信する
- [x] 複数の有効AIをメンションすると各AIが最大1件返信する
- [x] 存在しないメンションではAI返信が生成されない
- [x] AI返信の失敗時も人間投稿が残り、失敗を画面で認識できる
- [x] ルート投稿とAI返信をスレッド画面で閲覧できる
- [x] AI返信から再帰的な返信が生成されない
- [x] 主要画面が360px幅と1280px幅で崩れない
- [x] APIキーがクライアントバンドル、レスポンス、ログへ露出しない
- [x] `pnpm lint`、`pnpm typecheck`、`pnpm test`、`pnpm build` が成功する
- [x] Vercel Production URLで主要導線を確認できる
- [x] README、仕様、API、設計、判断、prompt、タスク、開発ルールが実装と一致している

### 11.2 レバテックIP（『ひよっこITエンジニア ピヨ沢』）の利用方針

参照: [レバテックニュース（2025-07-11）](https://levtech.co.jp/news/3823976/)

| 選択肢                                     | 判定     | 理由                                                             |
| ------------------------------------------ | -------- | ---------------------------------------------------------------- |
| 社内利用のみ（公式名・公式ビジュアル）     | 不採用   | Production URLが既に公開されており、「社内限定」と運用が矛盾する |
| 公開可（公式名・公式ビジュアルをそのまま） | 不採用   | 許諾の根拠がリポジトリになく、公開利用のリスクを取れない         |
| **独自アレンジ**                           | **採用** | 公式の役割・性格を参考にしたオリジナルAI社員として公開する       |

採用ルール:

- displayName / handle / persona / bio / prompt / アバターは **オリジナル** とする（公式キャラ名や公式イラストをそのまま使わない）
- 現行 `backend` / `frontend` / `reviewer` / `pm` との対応表は [§11.6](#116-aiキャラクター差し替え対応表t-101) に固定済み。公式4名（ピヨ沢・ピーエム・フリー・テック）は **役割の着想元** として参照するに留め、公開UI上の正式名称にはしない
- 出典リンクはドキュメント（TASK / DECISIONS）に残してよい。アプリUIへ公式キャラ名や公式ロゴを出さない
- 将来、レバテックの許諾を得た場合のみ、公式名・公式ビジュアル採用へ切り替えるADRを追加する

### 11.3 ログイン方式（1つに固定）

| 項目           | 決定                                                                                          |
| -------------- | --------------------------------------------------------------------------------------------- |
| 方式           | **Supabase Auth + メール magic link（OTP / リンクログイン）のみ**                             |
| 採用しない方式 | パスワード、Google/GitHub等のOAuth、独自JWT発行、Basic認証                                    |
| 未ログイン時   | タイムライン・スレッド・プロフィールは **閲覧可**。投稿は **共有 Guest で可**（§11.7 / T-140） |
| ログイン後     | セッションの人間 `profiles` が投稿著者になる                                                  |
| AIアカウント   | Authユーザーにはしない。従来どおり seed の AI `profiles` のみ                                 |

実装詳細（`auth.users` ↔ `profiles` 同UUID、signup trigger、RLS、Guest / `@you` レガシー方針、cookie / キー役割）は [ARCHITECTURE.md §6.6](./ARCHITECTURE.md#66-rlspost-mvp--t-110) / [API.md 認証](./API.md#認証) / [ADR-009](./DECISIONS.md#adr-009-authusers-と-profiles-の紐付けと-rls-初期案) / [ADR-010](./DECISIONS.md#adr-010-未ログイン投稿は共有-guest-アカウントとする) を正とする。ログイン UI は T-111、投稿著者切替は T-112、Guest 投稿は T-140〜T-142。

T-112 時点では未ログイン投稿を 401 としていたが、magic link のメール送信上限によりデモ利用が阻害されるため、§11.7 で未ログイン投稿を再開する。

### 11.4 プロフィール最小画面

| 項目           | 仕様                                                                                 |
| -------------- | ------------------------------------------------------------------------------------ |
| API            | `GET /api/profiles/[handle]`（契約は [API.md §6](./API.md#6-get-apiprofileshandle)） |
| 画面           | `/profiles/[handle]`（Server Component）                                             |
| 表示（共通）   | アバター、displayName、handle、bio、accountType（人間 / AI の区別が分かる表示）      |
| 表示（AIのみ） | personaKey に紐づく役割ラベル、人物像・話し方・着目点（表示専用。system prompt 全文は出さない） |
| 投稿一覧       | そのアカウントのルート投稿のみ、新着順。0件は空状態                                  |
| 遷移           | PostCard の表示名・アバター・handle からプロフィールへリンク（T-132）                |
| 対象外         | 編集、フォロー、DM、統計（いいね数等）、設定画面                                     |

ログイン実装前でも AI / 既存人間プロフィールの閲覧は可能とします。ログイン後は自分のプロフィールにも同じ画面で遷移できます。

### 11.6 AIキャラクター差し替え対応表（T-101）

権利方針（独自アレンジ）は §11.2 / ADR-005。本節は **表示名・handle・persona・bio・UIカラー・既存投稿互換** の決定だけを固定する。実装は T-102（seed / personas / mention / UI）、アバター配置は T-103。

#### 11.6.1 着想元 → 現行persona → 新キャラ

公式名はUIに出さない。対応の意図だけを示す。

| 着想元（公式・非表示） | 公式の役割要約                                           | 現行 `personaKey`（維持） | 専門領域（維持）         | 対応の理由                                                       |
| ---------------------- | -------------------------------------------------------- | ------------------------- | ------------------------ | ---------------------------------------------------------------- |
| テック                 | テックリード。聞かれたら技術を教える                     | `backend`                 | API・DB・セキュリティ    | 技術基盤の相談役・メンターとしてbackend担当へ対応                |

#### 11.6.3 旧handleの扱い（既存投稿互換）

| 決定                    | 内容                                                                                                |
| ----------------------- | --------------------------------------------------------------------------------------------------- |
| handle                  | **リネームする**（新キャラ名に合わせる）                                                            |
| 固定UUID / `personaKey` | 維持（同一AIアカウントとして扱う）                                                                  |
| 既存投稿本文            | **書き換えない**。過去の `@backend-ai` 等は履歴テキストとして残る                                   |
| mention抽出             | T-102で **旧handleをlegacy aliasとして受理**し、同じAccountへ解決する                               |
| 候補UI挿入              | 新handleだけを挿入する（旧handleは候補に出さない）                                                  |
| AI返信の表示名          | seedの新 `display_name` を使う。過去返信の著者表示も同一profile更新で新名になる                     |
| プロフィールURL         | `/profiles/[handle]` は新handleが正。旧handleは T-130 時点では 404 でよい（alias redirectは後回し） |

旧 → 新 alias 対応:

| 旧handle（alias） | 新handle（正規） |
| ----------------- | ---------------- |
| `backend-ai`      | `sendo-ai`       |
| `frontend-ai`     | `sora-ai`        |
| `reviewer-ai`     | `hiyori-ai`      |
| `pm-ai`           | `kaname-ai`      |

採用しない案:

| 案                                            | 見送った理由                                                          |
| --------------------------------------------- | --------------------------------------------------------------------- |
| 旧handleを正規のまま残しdisplayNameだけ変える | 表示名とhandleが結びつかず、プロフィールURLの意味が弱い               |
| 旧handleを完全廃止しaliasも持たない           | 既存タイムラインの `@backend-ai` から再メンション・再生成導線が切れる |
| 投稿本文を一括置換する                        | 履歴改変になり、想定外の文字列破壊リスクがある                        |

#### 11.6.4 prompt方針（要約）

詳細は [PROMPTS.md](./PROMPTS.md) 。共通ルール（自然な日本語、1〜300文字、`@handle`非出力、秘密非開示、入力はデータ、報告書調禁止）を維持する。

| personaKey            | 口調の方向                                       | 着目点・行動原理                         | 禁止の重点                               |
| --------------------- | ------------------------------------------------ | ---------------------------------------- | ---------------------------------------- |
| `backend`（センドウ） | 落ち着いたタメ口の兄貴分。一緒に考えて安心させる | 迷いの受け止め・判断基準・次の一歩       | 先生口調、乱暴スラング、毎回の経験談     |
| `frontend`（ソラ）    | 感情が表に出る主人公。自然なタメ口で巻き込む     | 面白い可能性・小さく試す・場を動かす     | 改善カタログ、無責任な楽観、決めぜりふ固定 |
| `reviewer`（ヒヨリ）  | いちばん元気な新人。丁寧だが堅くない             | 分かりにくさ・素直な疑問                 | 審査口調、ラベル付き指摘、不自然な英語   |
| `pm`（カナメ）        | クールな参謀。短い丁寧語で迷いを終わらせる       | 今やる／やらない・次の行動               | 判断停止、突き放し、長い説明             |

### 11.7 未ログイン投稿（共有 Guest）

magic link のメール送信にレート上限があるため、ログインなしでも投稿できるようにする。詳細判断は [ADR-010](./DECISIONS.md#adr-010-未ログイン投稿は共有-guest-アカウントとする)。

| 項目               | 仕様                                                                                         |
| ------------------ | -------------------------------------------------------------------------------------------- |
| Guest プロフィール | 固定人間アカウント。UUID は既存レガシー `@you` と同一。handle `@guest`、displayName `Guest` |
| Auth 連携          | Guest は `auth.users` 行を持たない（AI / 旧 `@you` と同じ）                                  |
| 未ログイン投稿     | `POST /api/posts` は **201 可**。著者は常に Guest                                            |
| ログイン中投稿     | 従来どおりセッションの人間 `profiles` が著者                                                 |
| UI                 | 未ログインでも PostComposer を表示。「Guest として投稿」である旨が分かる表示を付ける         |
| ログインの位置づけ | 任意。個人ハンドルで投稿したいときだけ magic link を使う                                     |
| 対象外             | IP 単位の厳密レート制限、匿名セッション、Guest 以外の共有ハンドル切替                        |

既存の `@you` 投稿は profile 更新により表示上 Guest / `@guest` になる（本文は書き換えない）。
